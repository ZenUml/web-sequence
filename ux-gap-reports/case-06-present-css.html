Auth gate count now at 4 — CSS tab is the latest

Clicking the CSS tab triggers the same generic "Welcome to ZenUML.com" login modal — identical to the modal that appears for PNG export, Share Link, and potentially other features. 4 separate features now use the exact same auth modal with zero context about why login is needed or what the user will gain by signing in.

Worse: keyboard focus was not trapped inside the login modal. Keyboard input typed while the modal was visible went into the underlying ZenUML editor, corrupting the diagram content. This is both a security-adjacient UX bug and an accessibility violation (WCAG 2.1 §2.1.2: No Keyboard Trap requires focus to stay in modal dialogs).

🔴 Identified UX Gaps

GAP-06-001
HIGH

Login modal does not trap keyboard focus — input leaks into underlying editor

When the auth modal appears (triggered by CSS tab click), keyboard focus is not constrained to the modal. Typing while the modal is visible sends keystrokes to the CodeMirror editor behind it, corrupting diagram content. Closing the modal then reveals broken ZenUML code — with no indication the editor was modified. This is an accessibility violation (WCAG 2.1 §2.1.2) and a data-integrity bug.

WCAG 2.1 §2.1.2 — No Keyboard Trap · Nielsen #5 — Error prevention
Implement proper focus trap in the modal: on open, move focus to the first interactive element; Tab/Shift+Tab must cycle within the modal; Escape closes and returns focus to the trigger element. Use a library like focus-trap or the native dialog element which traps focus by default.
GAP-06-002
HIGH

CSS tab is auth-gated with no prior indication — 4th feature to use generic login modal

The CSS tab sits next to the ZenUML tab in the editor header — appearing as an equal peer, always accessible. Clicking it silently triggers the generic "Welcome to ZenUML.com" auth modal. No lock icon, no disabled state, no tooltip, no pricing tier badge signals that CSS customization requires an account. Users who've been working for hours on a diagram have no forewarning that switching tabs will interrupt them.

Nielsen #1 — Visibility of system status · Progressive disclosure
Show a lock icon or "Pro" badge on the CSS tab for non-authenticated users. On click, show a contextual modal: "CSS customization is available to signed-in users. Sign in to style your diagrams with custom CSS." Alternatively, show a read-only CSS preview with example styles to demonstrate value before gating.
GAP-06-003
MEDIUM

Present button gives zero feedback — silently fails or opens invisible fullscreen

The "Present" button in the bottom bar has aria-label="Toggle Fullscreen" and title="Toggle Fullscreen Presenting Mode" — but the button label says only "Present". There is no visual feedback when clicked: no loading state, no fullscreen transition animation, no confirmation. In automation testing the button appeared completely non-functional. In real use, fullscreen mode may activate but without any transition or indicator that a mode change occurred.

Nielsen #1 — Visibility of system status
Add a brief visual transition when entering fullscreen (e.g. the diagram panel expands with a smooth animation). Show a visible "Exit Fullscreen" (Esc) affordance once in fullscreen mode. The button title and label should match: either "Present" or "Fullscreen", not both.
GAP-06-004
MEDIUM

Privacy badge (shield icon) is invisible unless users know to hover it

A shield-with-checkmark icon sits in the top-right corner of the diagram canvas. Hovering it reveals: "We (the vendor) do not have access to your data. The diagram is generated in this browser." This is a genuine trust signal — but it is entirely invisible unless users discover the icon and hover it. New users will never see this message. The privacy guarantee is the strongest selling point for enterprise use, yet it's buried in a hover tooltip on an unlabeled icon.

Nielsen #6 — Recognition over recall · Marketing alignment
Surface the privacy guarantee during onboarding (first load) as a brief callout or tooltip. Add a visible text label "Private" or "Local-only" next to the shield icon. Consider adding it to the footer or "About" section where users who care about data privacy actively look.
GAP-06-005
LOW

No way to restore the default example after deleting all editor content

If a user accidentally deletes all content (or over-applies Undo and clears the editor), there is no "Reset to example" or "Insert starter template" option. The editor becomes a blank black area. The toolbar insert buttons require existing content structure to work correctly. New users who erase the example while experimenting are stuck with an empty canvas and no guidance.

Nielsen #3 — User control and freedom · Nielsen #9 — Help users recover from errors
Add a "Load example" link that appears when the editor is empty (similar to VS Code's "Open Folder" empty state). Alternatively, show the starter example in a ghost/placeholder style when empty. A simple "New from template" option in the New button dropdown would also solve this.
