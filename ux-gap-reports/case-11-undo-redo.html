Data corruption confirmed: Cmd+Z on Page 1 undoes Page 2 edits — global undo stack not page-scoped

ZenUML uses a single shared CodeMirror editor instance for all pages. When switching pages, the editor content is replaced but the undo stack is not reset. This means pressing Cmd+Z on Page 1 after editing Page 2 will undo Page 2's edits while the Page 1 tab appears selected — corrupting content on the wrong page with zero warning.

Reproduction steps confirmed in live testing: (1) Type on Page 1 → (2) Add Page 2, type there → (3) Switch back to Page 1 → (4) Press Cmd+Z → Page 2's content appears in the editor with Page 1's tab selected → (5) Press Cmd+Z again → editor shows empty content while diagram canvas still renders Page 2's stale diagram.

🔴 Identified UX Gaps

GAP-11-001
HIGH

Global undo stack crosses page boundaries — Cmd+Z on Page 1 undoes Page 2 edits

ZenUML uses a single CodeMirror editor instance shared across all pages. Switching pages replaces the editor content but does not reset or checkpoint the undo history. As a result, pressing Cmd+Z on Page 1 after having edited Page 2 undoes Page 2's changes while the UI shows Page 1 as selected. A second Cmd+Z empties the editor entirely. The diagram canvas continues rendering stale Page 2 content, creating a split-brain state where the editor and diagram are out of sync. This is a data corruption bug — a user can silently overwrite or erase another page's content simply by pressing Undo on their current page.

Nielsen #5 — Error prevention · Nielsen #3 — User control and freedom · Data integrity
Each page's CodeMirror instance must maintain an independent undo history. Either: (a) create separate CodeMirror instances per page (one per tab), or (b) save and restore the full history state when switching between pages. VS Code, Notion, and Obsidian all maintain per-document undo histories — switching tabs never cross-contaminates undo stacks. Fix requires storing the full cm.historySize() state per page on tab switch.
GAP-11-002
HIGH

No warning before Cmd+Z silently erases entire page content

When the undo history reaches the initial state (before any content was entered), a single Cmd+Z can clear the entire editor to an empty state. In testing: starting from "A → B: hello" (2 lines), one Cmd+Z jumped back to the 18-line example, and one more cleared everything. There is no safeguard, no "Undo will erase all content on this page — continue?" confirmation, no count of how many undos remain before the page goes blank. Silent erasure of all work with no recovery path is one of the most damaging editor UX failures.

Nielsen #5 — Error prevention · Nielsen #9 — Help users recognize, diagnose, and recover from errors
Add a visual undo depth indicator (small counter near the editor, e.g. "⌘Z · 3 steps") so users know how close they are to the bottom of the stack. When the next Cmd+Z would clear all content, show a brief inline warning: "Press again to clear all content (no more undo history)." Sublime Text and VS Code both show "Can't undo further" as a status bar message when the stack is exhausted. Alternatively, never allow undo to go past the initial load state of a page.
GAP-11-003
MEDIUM

Toolbar inserts are not atomic undo units — require character-by-character reversal

When the toolbar "Participant" button inserts NewParticipant (13 characters), pressing Cmd+Z once removes the entire token — that part works. However, the newline inserted by the toolbar to separate the token from surrounding content requires an additional Cmd+Z, making the complete reversal of one toolbar action require 2+ undo steps. More critically: clicking a toolbar button first steals focus from the editor, meaning the user must re-click the editor before pressing Cmd+Z — or the undo may fire on the wrong context. The disconnect between toolbar action and undo granularity violates user mental models where "one action = one undo step."

Nielsen #7 — Flexibility and efficiency of use · Command pattern — each user action should be one undoable unit
Wrap each toolbar button click into a single CodeMirror transaction (cm.operation()) that groups all text insertions and cursor movements into one atomic undo entry. Ensure focus returns to the editor before the transaction is committed so Cmd+Z immediately after toolbar use reverses exactly the insert. GitHub's markdown toolbar and VS Code's snippet insertions are always single-undo-step operations.
GAP-11-004
MEDIUM

~2 second diagram re-render delay creates a decoupled undo experience

After pressing Cmd+Z, the editor content updates immediately (correct behavior). However, the diagram canvas takes approximately 2 seconds to re-render the undone state. During this window, the editor shows the undone text but the diagram still shows the superseded content — creating a contradictory view where both panels are simultaneously "right" but at different points in time. Users who glance at the diagram after pressing Cmd+Z will see stale content and may press Cmd+Z again, overshooting their intended undo target.

Nielsen #1 — Visibility of system status · Temporal consistency between coupled views
Show an explicit loading/re-rendering indicator on the diagram canvas during the 2-second render delay — a subtle spinner, pulsing border, or "Rendering…" label. This tells users "the diagram is catching up" rather than leaving them in a false steady state. Consider debounce optimization: if the re-render typically takes 500ms for small edits, the 2s delay for a 1-line undo suggests the full pipeline is being re-triggered rather than a diff-based update.
GAP-11-005
LOW

Redo shortcut (Cmd+Shift+Z) is undocumented — not listed in cheat sheet or any visible help

Redo works via Cmd+Shift+Z, which was confirmed in testing. However, this shortcut appears nowhere in the visible UI — not in the keyboard shortcuts panel, not in the cheat sheet, not in any tooltip. Users who know Cmd+Z for undo will typically try Cmd+Shift+Z (macOS standard) or Cmd+Y (Windows standard) for redo. The ZenUML cheat sheet panel lists multiple shortcuts but omits undo/redo entirely. A user who accidentally over-undoes cannot recover their work unless they already know the platform-specific redo shortcut from external knowledge.

Nielsen #6 — Recognition over recall · Documentation completeness
Add Cmd+Z (undo) and Cmd+Shift+Z (redo) to the keyboard shortcuts cheat sheet panel. Add tooltips to any undo/redo buttons if they exist (currently none are visible in the toolbar). Even a one-line mention in the "Help" sidebar would dramatically reduce the chance of users losing work by not knowing redo is available.
